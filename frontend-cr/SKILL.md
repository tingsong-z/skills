# 前端代码 CR 工具

对当前分支与 master 分支的代码变更进行 Code Review，生成结构化的 CR 报告。

## 适用场景

- 提交 PR 前的自查
- 团队 CR 前的预检
- 新人代码的质量把控

## CR 检查维度

以下 6 个维度覆盖前端代码质量的关键方面：

### 1. 逻辑正确性
- 状态更新是否使用了最新的 state/props（避免闭包陷阱）
- 异步操作的竞态条件（组件卸载后 setState、请求乱序返回）
- 条件渲染的分支是否完整（该处理的 loading/empty/error 态）
- null/undefined 安全检查（可选链 `?.`、空值合并 `??`）
- useEffect 依赖数组是否正确，是否有遗漏依赖
- 数字/金额计算是否有精度问题

### 2. 性能
- 不必要的大范围 re-render（缺少 useMemo/useCallback/React.memo）
- useEffect 中是否有未清理的订阅/定时器/事件监听
- 列表项 key 使用 index 或 unstable key
- 循环中调用 API（应批量请求）
- 大体积依赖引入（可替换为轻量方案或懒加载）
- 大列表未使用虚拟滚动（超过 100 项）
- 图片未做懒加载

### 3. 安全
- dangerouslySetInnerHTML / innerHTML 是否有 XSS 风险
- 用户输入是否直接拼接到 URL/跳转路径（开放重定向）
- 敏感信息（token、密钥、内网地址）是否硬编码
- 第三方脚本加载是否使用 integrity 校验
- postMessage 是否校验 origin
- 动态 import 的 URL 是否可能被注入

### 4. 代码规范
- 命名是否语义化（避免 a/b/c/data/temp/item 等无意义命名）
- 魔法数字/字符串是否应提取为常量
- 文件超过 300 行是否应拆分
- 函数超过 50 行是否应拆分
- 重复代码（同一逻辑出现 3 次以上）
- 注释掉的代码是否应删除
- console.log/debugger 是否误提交

### 5. 可维护性
- 过度的 prop drilling（超过 3 层考虑 Context 或组合）
- 组件职责是否单一（UI 渲染 + 数据请求混在一起）
- 硬编码的配置（API 地址、文字、颜色）是否应抽离
- 类型定义是否完整（避免 `any`、使用可选属性代替联合类型 `| undefined`）
- 状态管理选型是否合理（该放全局的放组件内，该放组件内的提到全局）
- 自定义 Hook 是否封装了可复用逻辑

### 6. 测试
- 新增的关键业务逻辑是否缺少单元测试
- 新增组件的核心交互是否缺少集成测试
- 边界条件是否有测试覆盖（空数据、极值、异常响应）
- 快照测试是否被滥用（只覆盖静态 UI 没逻辑）
- mock 是否过度导致测试失去意义

## 严重程度分类标准

### 必须修复（🔴 Critical / Major）
如果本次上线，以下问题必然或高概率导致 Bug/事故/线上故障：
- 逻辑错误（条件判断反了、useEffect 缺少 cleanup 导致内存泄漏）
- 安全漏洞（XSS、敏感信息泄露）
- 接口调用参数错误导致数据异常
- 状态更新时序问题（竞态条件）
- 异步错误未捕获导致白屏
- 业务关键路径的功能缺失

### 建议修复（🟡 Minor / Nice to have）
不影响功能但会降低代码质量和可维护性：
- 性能问题（缺少 memo、不必要的 re-render）
- 代码规范问题（命名、魔法数字、文件过长）
- 可维护性问题（prop drilling、职责混乱）
- 缺少必要的注释或文档
- 缺少测试但非关键路径

### 可不修复（🟢 Optional / Low priority）
个人偏好，不影响功能和可维护性：
- 风格偏好（箭头函数 vs function、export default vs named export）
- 不影响可读性的排版差异
- import 排序不统一
- 非关键路径的类型标注缺失
- 轻微代码异味（不影响理解）

## 特殊规则

- 如果变更文件 > 30个，聚焦核心变更文件（按变更行数排名前30%）
- CSS/Less/SCSS 文件只检查：z-index 层级管理、!important 滥用、响应式断点合理性
- package.json 变更检查：新增依赖的必要性、版本是否为最新稳定版、包大小
- 测试文件（*.test.*, *.spec.*, __tests__/*）降低优先级

## 执行流程

### 阶段 1：获取变更

```bash
# 1.1 确认当前分支名
git branch --show-current

# 1.2 获取 master 分支最新 commit
git merge-base HEAD origin/master
# 或
git merge-base HEAD master

# 1.3 获取变更文件列表
git diff origin/master --name-status
# 或
git diff master --name-status

# 1.4 获取变更统计
git diff origin/master --stat
# 或
git diff master --stat
```

如果 `origin/master` 不存在，尝试 `master` 或 `main`。

### 阶段 2：逐个文件 CR

对每个变更文件：
1. 使用 `git diff origin/master -- <文件路径>` 获取 diff
2. 读取当前文件内容理解上下文
3. 按 6 个维度逐项检查
4. 记录发现的问题：文件路径、行号、严重程度、类别、问题描述、修复建议

### 阶段 3：业务影响分析

对每个变更文件分析：
- **影响模块**：这个变更涉及哪个业务模块/页面/组件
- **影响范围**：直接影响哪些功能，间接影响哪些关联功能
- **风险等级**：高/中/低
- **回滚策略**：如果上线后出现问题，如何回滚（是否需要配置开关、数据迁移回滚等）
- **上下游依赖**：是否依赖后端接口变更、是否影响其他前端模块的数据格式
- **用户感知**：用户是否可见此变更，对用户体验的影响（正向/负向/不可见）

## 逐条问题详情

### 🔴 必须修复（3项）

#### CR-001 | 逻辑正确性 | src/pages/OrderList/index.tsx:42
**问题**：useEffect 中发起异步请求未处理组件卸载后的 setState
```tsx
// 当前代码
useEffect(() => {
  fetchOrders(status).then(setOrders);
}, [status]);
```
**问题所在**：组件卸载或 status 快速切换时，旧的请求返回后会覆盖最新数据，导致展示与筛选条件不匹配。
**修复建议**：
```tsx
useEffect(() => {
  let cancelled = false;
  fetchOrders(status).then(data => {
    if (!cancelled) setOrders(data);
  });
  return () => { cancelled = true; };
}, [status]);
```
**业务影响**：订单列表在用户快速切换 Tab 时可能展示错误数据，导致操作员对错误订单进行发货/退款等操作。

## 注意事项

1. **不要只看 diff**：diff 只显示变更的几行，需要读取文件理解上下文后给出判断
2. **优先检查业务关键路径**：涉及支付、订单、权限、数据的代码要重点检查
3. **不能确定的点标记出来**：依赖后端接口变更、涉及跨团队模块等无法确认的，在报告中标注"需人工确认"
4. **修复建议要可执行**：不要只说"优化性能"，要说"在 XX 行用 useMemo 包裹，依赖项是 [a, b]"
5. **组件文件优先**：先检查 .tsx 组件，再检查 .ts 工具/hooks，最后检查样式文件
